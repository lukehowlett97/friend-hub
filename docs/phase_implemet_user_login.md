# Phase: Implement Login Process

## Goal

Replace open account creation with a controlled login system where each real user has exactly one account.

The app should be easy for friends to use without passwords, while still preventing duplicate accounts and casual account hijacking.

The target model is:

* Admin-controlled user creation
* One account per friend
* Single-use invite codes
* PIN-based login
* Long-lived sessions
* Admin dashboard for user management
* Safe cleanup of old test users

---

## Core Decisions

### Passwords are off limits

Do not use normal passwords.

Friends should not need to remember a password. They should use a short PIN instead.

Recommended:

* 6-digit numeric PIN
* Stored as a secure hash
* Never shown back to anyone
* Protected by failed login limits

### Public signup should be removed

Users should not be able to create accounts themselves.

Only an admin can create users.

### Invite codes are for setup only

Invite codes should not be permanent login credentials.

They should be:

* Generated by an admin
* Tied to one specific user
* Single-use
* Stored as a hash
* Invalidated after use
* Shown only once to the admin

### PINs are for normal login

After claiming an invite, a user logs in with:

* Username
* PIN

### Sessions should last

Once logged in, users should stay logged in on that device.

They should only need the PIN again when:

* Using a new device
* Logging out
* Their session expires
* Their PIN is reset

---

## User Flow

### Admin creates a friend account

The admin opens the admin panel and creates a user.

Fields:

* Display name
* Username
* Role

Example:

Display name: Mike
Username: mike
Role: member

The system generates an invite code.

Example:

MIKE-RIVER-4829

The raw invite code is shown once.

The admin sends this code to Mike.

---

### Friend claims their invite

The user opens the app and chooses:

I have an invite code

They enter:

* Invite code
* New PIN
* Confirm PIN

If valid:

* Invite code is marked as used
* PIN hash is saved
* User is marked active
* Session is created
* User is logged in

---

### Returning user logs in

The user chooses:

I already have an account

They enter:

* Username
* PIN

If valid:

* Session is created
* Last login time is updated
* User is logged in

---

## Admin Account

There should be at least one admin account.

The admin can:

* Create users
* Generate invite codes
* See all users
* See account status
* Reset a user PIN
* Deactivate users
* Reactivate users
* Clean up old test users

Important:

The admin should not be able to see raw PINs.

The admin can see whether a PIN has been set, but not the PIN itself.

Reason:

* PINs should never be stored as plain text
* Showing PINs makes impersonation too easy
* Resetting a PIN is safer than viewing it
* This keeps the auth model clean

Admin can see:

* User exists
* User role
* User active status
* Whether PIN is set
* Whether invite is pending
* Whether account is locked
* Last login time

Admin should not see:

* Raw PIN
* PIN hash
* Invite code hash
* Session tokens

## Auth Endpoints

## Create user

Admin-only.

Endpoint:

POST /api/v1/admin/users

Input:

* display_name
* username
* role

Behaviour:

* Require admin session
* Validate username is unique
* Create user
* Generate invite code
* Store invite code hash
* Return raw invite code once

Response includes:

* user
* invite_code

The invite code should never be retrievable again after this response.

---

## Claim invite

Public endpoint.

Endpoint:

POST /api/v1/auth/claim-invite

Input:

* invite_code
* pin
* pin_confirm

Behaviour:

* Validate invite code
* Check invite code has not expired
* Check invite code has not already been used
* Validate PIN format
* Hash PIN
* Save PIN hash
* Mark invite code as used
* Clear or invalidate invite code hash
* Mark user active
* Create session
* Return current user

PIN validation:

* PIN must be numeric
* PIN must be 6 digits
* PIN and confirmation must match

---

## PIN login

Public endpoint.

Endpoint:

POST /api/v1/auth/pin-login

Input:

* username
* pin

Behaviour:

* Find active user
* Check account is not locked
* Verify PIN hash
* Reset failed login count on success
* Update last login time
* Create session
* Return current user

Failure behaviour:

* Return a generic error
* Do not reveal whether the username or PIN was wrong
* Increment failed login count if user exists
* Lock account briefly after repeated failures

Generic error message:

Login failed. Check your details and try again.

---

## Current user

Endpoint:

GET /api/v1/auth/me

Behaviour:

* Return current logged-in user
* Include role
* Include whether user is admin

---

## Logout

Endpoint:

POST /api/v1/auth/logout

Behaviour:

* Delete or invalidate current session
* Clear session cookie

## Admin Endpoints

## List users

Admin-only.

Endpoint:

GET /api/v1/admin/users

Return:

* id
* username
* display_name
* role
* is_active
* has_pin
* invite_pending
* invite_code_used_at
* invite_code_expires_at
* locked_until
* failed_login_count
* created_at
* last_login_at

Do not return:

* pin_hash
* invite_code_hash
* session tokens
* raw PINs

---

## Reset PIN

Admin-only.

Endpoint:

POST /api/v1/admin/users/{user_id}/reset-pin

Behaviour:

* Require admin
* Clear existing PIN hash
* Generate new single-use invite code
* Store invite code hash
* Clear invite_code_used_at
* Set invite_code_expires_at
* Invalidate existing sessions
* Return raw invite code once

This lets the user set a new PIN.

---

## Deactivate user

Admin-only.

Endpoint:

POST /api/v1/admin/users/{user_id}/deactivate

Behaviour:

* Set is_active to false
* Invalidate active sessions
* Keep related messages and items intact

Do not hard-delete the user from the database unless it is safe.

---

## Reactivate user

Admin-only.

Endpoint:

POST /api/v1/admin/users/{user_id}/reactivate

Behaviour:

* Set is_active to true
* Optionally generate a new invite code
* Optionally force PIN setup again

---

## Change role

Admin-only.

Endpoint:

PATCH /api/v1/admin/users/{user_id}/role

Behaviour:

* Allow role changes between admin and member
* Prevent removing the final active admin account

Important rule:

There must always be at least one active admin.

---

## Sessions

Prefer HTTP-only cookies.

Session behaviour:

* Create session on invite claim
* Create session on PIN login
* Store only hashed session tokens in the database
* Cookie should be HTTP-only
* Cookie should be Secure in production
* Cookie should use SameSite Lax or Strict
* Session expiry can be generous for a private mates app

Recommended session duration:

30 to 90 days

Sessions table:

sessions

* id
* user_id
* session_token_hash
* expires_at
* created_at
* last_seen_at
* user_agent
* ip_address optional

---

## Rate Limiting and Lockout

PINs are short, so brute-force protection matters.

Recommended rule:

5 failed attempts locks the account for 10 minutes.

Behaviour:

* Increment failed_login_count after wrong PIN
* Set locked_until after threshold
* Reset failed_login_count after successful login
* Return generic errors

Avoid exposing exact failure reasons.

Bad:

Wrong PIN for Mike

Better:

Login failed. Check your details and try again.

## Database Cleanup

Need a one-off cleanup process for old testing users.

Goal:

* Keep admin account
* Keep real friend accounts
* Remove or deactivate duplicate testing accounts
* Avoid breaking messages, items, photos, events, comments, notifications, and foreign keys

Preferred approach:

Deactivate old test users rather than hard-delete them.

For deleted or deactivated users:

* Keep their old messages visible
* Show their display name if safe
* Otherwise show Deleted User

Possible cleanup modes:

## Safe mode

* Deactivate selected test users
* Invalidate their sessions
* Keep all content linked

## Anonymise mode

* Deactivate selected test users
* Rename display name to Deleted User
* Remove avatar or profile data
* Keep messages and content

## Hard-delete mode

Only use if foreign keys and cascading behaviour are fully understood.

Do not use hard delete as the default.

## Security Notes

### Do not store raw PINs

PINs must be hashed.

Use a proper password hashing function even though these are PINs.

Good options:

* argon2
* bcrypt
* passlib with bcrypt

### Do not log secrets

Never log:

* PINs
* Invite codes
* Session tokens
* Cookie values

### Do not expose hashes

Never return these in API responses:

* pin_hash
* invite_code_hash
* session_token_hash

### Protect admin endpoints

Every admin endpoint must check:

* User is authenticated
* User role is admin
* User is active

### Prevent final admin removal

Do not allow:

* Deactivating the final active admin
* Changing the final active admin to member
* Deleting the final active admin

After this phase:

* Nobody can freely create accounts
* Admin controls all users
* Each friend has one account
* Friends use PINs, not passwords
* Invite codes are single-use
