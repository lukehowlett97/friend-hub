Phase: Tagging, Backlinks, Item History, and Archive System
Goal

Improve the app’s information structure so every major item can be tagged, linked, referenced from chat, audited, and safely archived instead of permanently deleted.

This phase makes the app feel less like separate pages and more like one connected Hub.

The core idea:

Every item should have tags, backlinks, history, and a safe delete path.

Items include:

events
polls
ideas
reminders
photos
chat-created items
council motions
future rules and roles
Core Concepts
1. Tags

Tags help users find and group items.

Examples:

beans
pub
holiday
council
funny
important
2026-05-08
source-chat
source-event
event-bean-party

Tags should be searchable, clickable, and consistent across all item types.

2. Backlinks

Backlinks show how items are connected.

Examples:

a photo belongs to an event
a poll was created from chat
a reminder was created from an event
an idea spawned a poll
a chat message mentioned an item
a council motion changed a member role

This should not rely only on loose text tags. The app should store proper item-to-item relationships.

3. Item History

Every meaningful change to an item should be recorded.

This should work like a simple Git history.

Examples:

item created
title changed
description changed
tags changed
comment added
reaction added
poll option added
event time changed
item pinned
item unpinned
item archived
item restored
item deleted permanently, if that is ever allowed

Each history entry should record:

item id
item type
action type
changed by
timestamp
previous value, where useful
new value, where useful
summary message

Example history entry:

Luke changed the event title from “Bean Night” to “Bean Party”.

4. Archive / Recycle Bin

Deleting an item should not permanently remove it.

Instead, deleted items should move to an archive.

Archived items should:

disappear from normal lists by default
remain searchable from the archive
keep their tags
keep their comments
keep their history
keep their backlinks
be restorable by an admin or owner

This prevents accidental loss and makes the app feel safer.

Feature 1: Standardised Tags

All major items should use the same tag structure.

Applies to:

events
polls
ideas
reminders
photos
hub items
council motions

Tags should be stored as a list of normalised strings.

Normalisation rules:

trim whitespace
remove empty tags
deduplicate tags
handle leading hash consistently
avoid weird casing issues
reject unsafe characters
keep display labels clean

Example:

Input:

Beans
beans
#Pub
empty value
Friday Night

Stored as:

beans
pub
friday-night
Feature 2: System Tags and User Tags

Tags should eventually be split into two categories.

System Tags

Generated by the app.

Examples:

type-event
type-photo
date-2026-05-08
source-chat
source-event
created-by-luke
parent-event-bean-party

System tags should not usually be manually removable.

User Tags

Added by users.

Examples:

funny
pub
beans
holiday
important
chaos

User tags can be edited normally.

For the first implementation, it is acceptable to store all tags together, but the code should be structured so system tags and user tags can be separated later.

Feature 3: Photo Auto-Tagging

Photos should receive automatic tags based on where they were uploaded.

Photo uploaded from an event:

type-photo
source-event
date-2026-05-08
event-bean-party
uploaded-by-luke

Photo uploaded from chat:

type-photo
source-chat
date-2026-05-08
uploaded-by-luke

Photo uploaded from a poll:

type-photo
source-poll
date-2026-05-08
poll-pub-vote
uploaded-by-luke

This makes photos easier to find later.

Feature 4: Explicit Backlinks

Add a proper relationship model between items.

Each relationship should contain:

source item id
source item type
target item id
target item type
relationship type
created by
created at

Relationship types:

attached to
created from
mentioned in
belongs to event
spawned poll
spawned reminder
references
discussed in
voted on by
archived with
restored from

Examples:

Photo belongs to Event
Poll created from Chat Message
Reminder created from Event
Idea spawned Poll
Chat Message references Poll

This gives the app real backlinks rather than relying on tags alone.

Feature 5: Chat-Callable Item References

Every Hub Item should have a short reference.

Examples:

E-3 for an event
P-12 for a poll
I-7 for an idea
R-4 for a reminder
PH-18 for a photo
M-2 for a council motion

When someone types a reference in chat, the app should render a small preview card.

Example chat message:

Anyone going to E-3?

Rendered preview:

Event: Bean Party
Friday 8 May, 11:35
View event

This should make the chat feel connected to the rest of the app.

Feature 6: Item History

Every major item should have a history timeline.

History should record important changes.

Actions to track:

created
updated
renamed
description changed
tags changed
date changed
attendees changed
poll vote cast
poll closed
comment added
comment edited
comment deleted
reaction added
reaction removed
pinned
unpinned
sent to chat
linked to another item
unlinked from another item
archived
restored

Each history entry should include:

action type
actor
timestamp
human-readable summary
before value, where useful
after value, where useful
related item, where useful

Example:

Luke changed tags from “beans” to “beans, party, council”.

Example:

Hannah archived Poll P-12.

Example:

Mike restored Event E-3 from the archive.

Feature 7: History UI

Each item detail page should have a History tab or section.

Possible tabs:

Details
Comments
Links
History

The History section should show a simple timeline.

Example:

Today 12:44
Luke changed the title from “Bean Night” to “Bean Party”.

Today 12:40
Gang added tag “beans”.

Today 12:35
Gang created the event.

Keep it readable and not too technical.

Feature 8: Archive / Recycle Bin

Deleting an item should move it to the archive.

Archived items should have:

archived_at
archived_by
archive_reason, optional
restored_at, optional
restored_by, optional

Archived items should not appear in normal views unless the user chooses to show archived items.

Archive views:

All archived items
Archived events
Archived polls
Archived ideas
Archived reminders
Archived photos

Actions:

restore item
view history
permanently delete, admin only and optional

Recommended default:

Do not implement permanent delete yet.

Feature 9: Comments and Edits in History

Comments should also appear in item history.

Examples:

Luke commented on Event E-3
Mike edited a comment
Hannah deleted a comment
Joe reacted with laughing emoji

This makes the activity feed useful later.

Eventually, the app could have a global activity page:

Recent Hub Activity

Luke created Event E-3
Hannah voted on Poll P-12
Mike commented on Idea I-4
Gang archived Reminder R-2
Feature 10: Hub Bot Integration

Hub Bot should be able to announce important item changes in chat.

Examples:

Hub Bot: Event updated
Bean Party now starts at 12:00.

Hub Bot: Poll archived
Pub? was moved to the archive by Luke.

Hub Bot: Item restored
Bean Party has been restored from the archive.

Hub Bot should not announce every tiny edit by default, or the chat will become noisy.

Good Hub Bot announcements:

new item created from chat
item sent to chat
poll closed
event starting soon
item archived
item restored
important date changed
council decision applied

Quiet history only:

minor tag edit
typo fix
reaction added
comment edited
Recommended Build Order
Phase 1: Fix and Standardise Tags

Build:

fix event tag editing bug
shared tag normalisation
consistent tag field across events, polls, ideas, reminders, photos, and hub items
tag chips in the UI
regression tests

This should be done first because the current tag edit behaviour is unreliable.

Phase 2: Archive Instead of Delete

Build:

archived_at field
archived_by field
archive endpoint
restore endpoint
archive page
hide archived items from normal lists
update delete buttons to archive instead

This gives immediate safety.

Phase 3: Item History

Build:

item history table
history logging service
log create, update, archive, restore, comment, and tag changes
history UI on item detail pages

This gives the Git-style audit trail.

Phase 4: Backlinks

Build:

item relationships table
link and unlink endpoints
backlinks section on item detail pages
automatic links from photos, chat references, polls, and events

This gives the app its connected knowledge graph.

Phase 5: Chat References and Previews

Build:

parse item references in chat
render preview cards
allow opening the linked item
record mentioned in relationships

This makes Hub Items callable from chat.

Phase 6: Hub Bot Activity Layer

Build:

Hub Bot announcements for important item changes
quiet history for minor edits
chat messages for archive, restore, poll close, event update, and item creation
future natural language commands
